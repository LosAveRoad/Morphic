# Cognitive Lab 前端 MVP 规格

**日期**: 2026-04-18
**作者**: GPT-5.4
**状态**: 待审阅
**目标**: 一次性完成一个可演示的前端 MVP，在不依赖真实后端的前提下展示产品核心交互

## 1. 目标

构建一个基于 `tldraw` 的前端 MVP，用最小但完整的闭环呈现 Cognitive Lab 的核心体验：

- 沉浸式 Notion 风极简画布
- Apple Pencil / 手写友好的无限画布交互
- 通过 `+` 锚点进入 AI 交互
- 使用本地 mock 数据完成推荐与内容生成
- 在画布上插入、拖拽和排布生成卡片
- 通过 redo 切换到另一种结果形态

该 MVP 的目标不是“功能齐全”，而是“产品感觉真实、交互闭环完整、后续接入真实 API 时不需要重做前端骨架”。

## 2. 非目标

本次 MVP 不包含以下内容：

- 不接入真实后端 API
- 不实现登录、注册、用户偏好设置
- 不实现云端画布保存与多端同步
- 不实现复杂的学习原语库、图表引擎、全局知识图谱
- 不实现 Apple Pencil 靠近感应等硬件高级能力
- 不实现真实 AI 推理，仅用本地 mock 结果驱动 UI

## 3. 产品形态

MVP 只有一个主工作台页面，打开后直接进入画布，不增加额外 landing page 或多路由切换。

整体风格：

- 极简、轻量、偏 Notion
- 浅色背景、克制边框、弱阴影、大量留白
- 页面工具感弱，画布存在感强
- AI 入口轻量化，不做大型聊天面板

MVP 必须完整呈现以下四个演示瞬间：

1. 用户点击 `+` 号后弹出推荐项
2. 用户输入一句话后生成卡片
3. 用户能在画布上拖拽和重新排布卡片
4. 用户点击 redo 后看到“另一种结果”而不是重复结果

## 4. 核心页面结构

MVP 采用单页结构，由以下区域组成：

### 4.1 顶部极简栏

负责：

- 显示产品标题或轻量状态
- 提供少量全局操作，例如清空演示、切换帮助提示

不负责：

- 承载复杂 AI 配置
- 承载多级导航

### 4.2 主画布区

负责：

- 渲染 `tldraw`
- 提供手写、平移、缩放和基础空间交互
- 作为 AI 锚点和生成卡片的空间容器

### 4.3 AI 锚点与面板

负责：

- 在当前上下文位置出现 `+` 锚点
- 打开推荐项与直接输入面板
- 触发本地 mock 推荐和本地 mock 生成

### 4.4 卡片层

负责：

- 在画布坐标系中放置生成卡片
- 支持拖拽、选中、重排和 redo
- 呈现文本卡、提纲卡、混合卡

### 4.5 轻量反馈层

负责：

- 呈现生成中、redo 中、保存中等轻量状态
- 提供最小操作反馈，不打断主画布体验

## 5. 技术方案

### 5.1 核心栈

MVP 保持以下技术方向：

- `Next.js`
- `React`
- `TypeScript`
- `tldraw`
- `Tailwind CSS`
- 本地状态管理工具，可优先使用 `Zustand`

### 5.2 为什么首版必须使用 tldraw

虽然仅用自由拖拽画布层可以更快，但本产品的核心价值之一就是 Apple Pencil 和自然书写体验，因此 MVP 必须从第一版开始建立 `tldraw` 画布基础。

这带来的好处是：

- 提前锁定画布坐标系统
- 提前验证 iPad / 手写的产品感觉
- 后续接真实 AI 与真实持久化时不需要重建空间容器

### 5.3 为什么 AI 先使用 mock

当前后端未完成，不适合把前端主路径阻塞在接口联调上。首版先用 mock，可以优先完成：

- 锚点交互
- 推荐面板
- 输入生成
- 卡片插入
- redo 切换

但代码结构上仍需保留 `AI Provider` 边界，保证后续切换真实 API 时只替换数据来源，而不重写 UI。

## 6. 领域边界

前端 MVP 至少拆成以下领域：

### 6.1 Canvas Domain

负责：

- 初始化和承载 `tldraw`
- 获取当前视口、点击位置、基础上下文信息
- 提供卡片锚定坐标

### 6.2 AI Domain

负责：

- 管理锚点、推荐面板、输入框、生成状态
- 通过本地 mock provider 返回推荐项和生成结果
- 支持 redo 切换结果

### 6.3 Card Domain

负责：

- 维护卡片数据结构
- 控制卡片插入、拖拽、移动、选中
- 支持卡片类型渲染

### 6.4 Persistence Domain

负责：

- 本地保存卡片状态
- 本地恢复当前画布中的卡片数据
- 与 `tldraw` 本地持久化并存，但不混成一个黑盒

## 7. 核心交互流程

### 7.1 锚点推荐流程

1. 用户点击画布空白区域，或在当前焦点附近触发 AI 锚点。
2. `+` 锚点出现，并带轻微呼吸感。
3. 用户点击锚点后，弹出推荐面板。
4. 前端从本地 mock provider 取出 3-4 个推荐项。
5. 用户可以点推荐项，也可以直接输入一句话。

### 7.2 生成卡片流程

1. 用户选择推荐项，或输入一句话。
2. 前端调用本地 mock generation 逻辑。
3. 返回一张或多张内容卡片的数据结构。
4. 卡片插入到锚点附近的画布坐标中。
5. 用户可以立刻拖动并重新排列卡片。

### 7.3 Redo 流程

1. 用户点击某张卡片上的 redo。
2. 系统不重复上一结果，而是从该卡片的候选结果池里切换到另一种变体。
3. 卡片内容更新，视觉上出现简短过渡。

## 8. AI Mock 设计

### 8.1 推荐项

首版推荐项为本地静态模板，但允许根据用户输入和位置上下文做轻量分支。推荐项类型例如：

- 解释这个概念
- 整理成提纲
- 生成练习题
- 换一种表达

### 8.2 生成结果

首版至少支持以下卡片结果：

- `text`
  - 简短解释型卡片
- `outline`
  - 提纲或要点卡片
- `hybrid`
  - 说明 + 小块结构化内容

### 8.3 Redo 候选池

每个生成动作都要对应至少两种可切换结果，例如：

- 简洁解释
- 提纲整理
- 问题引导

redo 的核心是“换一种表达方式”，而不是随机刷新。

## 9. 卡片模型

每张卡片至少包含以下字段：

```ts
type CanvasCard = {
  id: string
  x: number
  y: number
  width: number
  height: number
  type: 'text' | 'outline' | 'hybrid'
  title?: string
  content: unknown
  variantKey: string
  redoOptions: string[]
}
```

要求：

- 所有卡片能在画布上独立拖拽
- 所有卡片位置以画布坐标管理
- redo 不改变卡片身份，只切换其内容和 `variantKey`

## 10. 持久化策略

MVP 只做本地持久化，不做云端同步。

需要区分两类本地状态：

- `tldraw` 负责的画布本地状态
- 应用自己负责的卡片本地状态

页面刷新后应尽量恢复：

- 已存在的卡片
- 卡片位置
- 最近一次基本布局状态

不要求恢复每一次临时推荐弹层或输入框状态。

## 11. 目录建议

前端工程建议按以下结构组织：

```text
src/
  app/
    page.tsx
    layout.tsx
    globals.css
  components/
    shell/
    canvas/
    ai/
    cards/
  domains/
    ai/
    cards/
    persistence/
  lib/
    mock/
    utils/
  stores/
    uiStore.ts
    cardStore.ts
```

可以在实现时略有调整，但不要让 AI、卡片、画布逻辑全部堆在一个文件里。

## 12. 验收标准

当以下条件全部满足时，MVP 视为完成：

- 能启动一个可访问的前端页面
- 页面默认呈现 Notion 风极简工作台
- 中央为真实 `tldraw` 画布
- 用户可通过 `+` 号看到推荐项
- 用户可输入一句话并生成本地 mock 卡片
- 卡片可在画布上拖拽与排布
- redo 能切换出另一种结果
- 刷新后卡片基本状态可以恢复
- 整个流程不依赖真实后端

## 13. 后续演进

当 MVP 稳定后，可沿以下方向继续推进：

- 将 AI provider 从 mock 切换为真实 API provider
- 将本地卡片持久化切换为画布 API 保存
- 增加更丰富的卡片类型和学习原语
- 增强锚点动画与 redo 过渡动画
- 深化 iPad PWA 与 Apple Pencil 体验
