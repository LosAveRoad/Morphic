# Cognitive Lab 前端架构设计

**项目名称**: Cognitive Lab Frontend
**文档类型**: 前端架构与边界设计
**创建日期**: 2026-04-18
**版本**: 2.0
**状态**: 设计阶段

---

## 1. 总览与产品目标

Cognitive Lab 前端不是一个通用白板应用，而是一个面向 iPad 和桌面 Web 的沉浸式无限画布学习界面。前端设计的重点不是堆叠功能，而是围绕“纸张感画布、轻量 AI 锚点、结构化生成内容、可持续同步”四类体验建立稳定边界。

首期前端目标：

- 提供沉浸式、低干扰的无限画布体验
- 以锚点驱动 AI 交互，而不是以聊天面板驱动交互
- 以受控内容渲染承接文本、卡片和混合结果
- 与现有后端 API 契约对齐，不要求后端额外改动
- 兼顾 iPad PWA 和桌面 Web 的一致性体验

前端最终扮演的角色不是“AI 结果展示页”，而是“沉浸式画布中的交互编排器”。

---

## 2. 前端范围与非目标

### 2.1 前端范围

本阶段前端负责：

- 画布界面、视口交互、手写输入与布局体验
- AI 锚点触发、推荐选项展示、生成流程编排
- 文本、结构化卡片、混合内容的受控渲染
- 画布持久化、本地恢复与离线兜底
- 与现有认证、画布、AI API 的契约对接

### 2.2 非目标

本阶段前端不负责：

- 修改后端 API 设计或新增后端字段
- 重新定义模型策略、推荐排序或 Prompt 逻辑
- 在首期就完整实现所有未来学习原语和外部平台接入
- 将 HTML 内容当作不受限制的网页直接执行

---

## 3. 核心体验原则

首期前端体验遵循以下原则：

- 画布优先：默认隐藏高干扰工具元素，保留沉浸式纸张感
- 锚点驱动：AI 通过上下文锚点进入，而不是抢占主界面
- 结果受控：所有 AI 生成内容必须进入受控渲染容器
- 重构可感：redo 不只是重试，而应呈现“坍缩并重构”的状态过渡
- iPad 优先：关键交互以 iPad PWA 的触控和手写体验为基线

与 `main.md` 对齐的首期体验重点包括：

- 全屏、低工具干扰的画布外观
- 可感知的 AI 锚点浮现与轻微呼吸感
- 支持文本式和卡片式两种主要结果落地方式
- 允许未来演进到更丰富的学习卡片与实验性组件

## 4. 前端领域边界

前端按五个领域拆分，每个领域只负责一类能力。

### 4.1 Canvas Surface

负责：

- 渲染 `tldraw` 画布
- 处理视口、绘制、选区和 Apple Pencil 输入
- 收集附近内容、选区信息和画布上下文
- 呈现沉浸式视觉外壳

不负责：

- 决定 AI 推荐逻辑
- 管理 AI 会话生命周期
- 直接渲染后端返回的原始 HTML 载荷

### 4.2 AI Interaction Flow

负责：

- 管理锚点浮现与交互状态机
- 调用推荐接口与生成接口
- 协调选项展示、直接输入、redo 和 morphing 过渡
- 维护用户可见的 AI 过程状态

不负责：

- 原始画布持久化
- 生成内容内部结构的安全渲染
- 后端推理策略

### 4.3 Generated Content Renderer

负责：

- 安全渲染 `text`、`html`、`hybrid` 三类内容
- 将内容映射为文本块、结构化卡片或交互卡片
- 为卡片结果提供统一的尺寸、布局和插入规则

不负责：

- 发起 AI 请求
- 持有认证状态
- 保存会话状态

### 4.4 Session and Persistence

负责：

- 管理 `sessionId`、当前推荐选项、最近一次交互上下文
- 组织本地恢复和草稿保护
- 编排防抖保存和离线兜底逻辑

不负责：

- 替代后端成为唯一真相来源
- 将业务语义硬编码到缓存结构中

### 4.5 API Integration Boundary

负责：

- 将前端视图模型映射到后端请求与响应契约
- 集中处理认证头、错误转译和重试语义
- 为上层 UI 提供领域化 API 能力

不负责：

- 发明新的后端字段
- 将所有前端逻辑塞进单一巨型 `ApiClient`

### 4.6 推荐的目录边界

文档层面建议前端代码按能力组织，而不是按“所有 hooks 放一起、所有 utils 放一起”的方式无限扩张：

```text
src/
  app/
  domains/
    canvas/
    ai/
    rendering/
    persistence/
    auth/
  lib/
    api/
      transport/
      authApi.ts
      canvasApi.ts
      aiApi.ts
  stores/
    uiStore.ts
    sessionStore.ts
  styles/
```

这只是边界建议，不要求一次性完成大规模重构。

## 5. 标准 AI 交互流程

MVP 的标准 AI 流程定义为：

`idle -> sensing -> options-ready -> generating -> rendered -> morphing-redo`

具体行为如下：

1. 用户在画布中停留、书写或聚焦某一区域。
2. 上下文 AI 锚点变得可用。
3. 前端提取 `canvasContext` 并调用 `POST /recommend-options`。
4. 后端返回 `sessionId` 和推荐选项。
5. 用户选择推荐项，或直接输入自己的需求。
6. 前端使用同一个 `sessionId` 调用 `POST /generate-content`。
7. 返回结果交由受控渲染层处理。
8. 渲染后的结果通过画布保存链路写回画布文档。

推荐的前端编排方式：

```typescript
const options = await aiApi.recommendOptions(canvasContext)

const content = await aiApi.generateContent({
  sessionId: options.sessionId,
  selectedOptionId,
})
```

需要特别强调：

- 不再将单步 `/generate` 视为主路径
- `sessionId` 属于 AI 交互过程，而不是画布身份
- redo 的职责是“重新组织生成路径”，不是简单重复上一次请求

## 6. API 集成边界

前端以 `docs/api-reference.md` 为唯一后端契约参考，遵循以下规则：

- AI 推荐入口为 `POST /recommend-options`
- 内容生成入口为 `POST /generate-content`
- `sessionId` 仅表示 AI 交互会话，不等同于画布 ID
- 生成内容类型以 `text`、`html`、`hybrid` 为准
- 前端可以定义本地适配层，但不能要求后端提供未文档化字段

建议将 API 层拆分为两层：

- 传输层：负责请求发送、认证头、超时、错误转译
- 领域 API 层：例如 `authApi`、`canvasApi`、`aiApi`

推荐职责如下：

- `authApi`：登录、注册、登出、当前用户信息
- `canvasApi`：画布创建、获取、列表、删除、保存
- `aiApi`：推荐选项、生成内容、会话级错误处理

认证错误和速率限制不应散落在 UI 组件中处理，而应先在领域 API 层归一，再向界面暴露明确状态。

## 7. 渲染边界

生成结果必须通过受控渲染层落地，按以下三级容器划分：

- `Inline Text Block`
  - 用于解释、总结、笔记等文本输出
- `Structured Card`
  - 用于有排版层次的结构化内容
- `Interactive Card`
  - 用于受控 HTML 或混合内容

对 `html` 和 `hybrid` 内容，前端必须保证：

- 不执行脚本
- 不允许无控制的全局样式注入
- 不允许原始内容直接携带隐式事件处理器
- 所有交互都包裹在前端受控容器中

因此，前端文档不应再默认使用“后端给什么 HTML 就直接插入页面”的描述方式。更稳妥的方式是：

- 文本内容优先走 Markdown 或结构化文本渲染
- 卡片内容进入受控卡片壳层
- 混合内容通过统一的布局容器组合文本区与交互区

首期推荐的主力内容形态是 `hybrid`，因为它既能保留学术表达，也能承载局部交互。

## 8. 持久化与离线策略

前端需要明确区分三类状态。

### 8.1 画布持久化

- 存储画布元素、卡片位置和用户最终可见结果
- 与后端画布 API 同步
- 作为跨设备与长期保存的正式文档层

### 8.2 AI 会话状态

- 存储 `sessionId`、推荐选项和 redo 相关临时上下文
- 与画布持久化分离
- 会话状态可以被重置，但不能破坏画布文档本身

### 8.3 本地恢复缓存

- 提供崩溃恢复和离线兜底能力
- 可以利用 `tldraw` 本地持久化作为实现手段
- 不能取代后端成为最终真相来源

推荐的优先级关系：

- 后端画布保存结果：正式持久化真相
- 本地恢复缓存：短期恢复与异常兜底
- AI 会话状态：交互期临时状态

这意味着 `tldraw persistenceKey` 可以保留，但只能被定义为“本地恢复机制的一部分”，不能被写成最终持久化故事。

## 9. 技术建议

本阶段建议保留当前总体技术方向：

- `Next.js` + `React` + `TypeScript`
- `tldraw` 作为无限画布引擎
- `Tailwind CSS` 作为样式基础
- `Zustand` 仅用于聚焦共享状态
- PWA 作为 iPad 安装与离线能力基础

同时建议做以下约束：

- `Zustand` 只承载轻量共享状态，不统管全部业务
- API 逻辑拆层，不把请求、副作用、跳转和业务判断都塞进一个客户端对象
- `tldraw` 不服务端渲染，重型画布能力按需懒加载
- OCR 能力视实际性能表现决定是否走本地或异步路径，不阻塞主交互流

本阶段的重点不是更换技术栈，而是建立清晰边界。

## 10. 分阶段实现优先级

### Phase 1: 画布基础与体验外壳

- 搭建 Next.js 前端基础框架
- 集成 `tldraw` 并建立沉浸式全屏画布
- 完成 PWA 配置和 iPad 安装体验
- 建立最小化工具干扰的视觉基线

### Phase 2: 数据与会话骨架

- 接入认证 API 与画布 API
- 完成画布加载、保存、本地恢复
- 建立 `sessionId` 驱动的 AI 会话状态

### Phase 3: AI 交互主路径

- 实现 AI 锚点触发
- 接入 `POST /recommend-options`
- 接入 `POST /generate-content`
- 完成推荐选项、直接输入和渲染主路径

### Phase 4: 内容渲染与重构体验

- 建立 `Inline Text Block`、`Structured Card`、`Interactive Card`
- 加入 redo 的 morphing 过渡体验
- 优化 iPad 手写与内容插入的连贯性

### Phase 5: 质量与扩展

- 强化错误处理、限流提示和离线兜底
- 增加交互测试与主路径 E2E
- 为后续学习卡片、模拟器和实验性组件保留扩展接口
